# CREDO-011 - Librería @one/credo para Estandarizar Uso de Credo

## Descripcion de la Tarea

Crear una libreria compartida que unifique la forma en que issuer-service, holder-service y verifier-service usan Credo-TS. Actualmente existe duplicacion significativa de adaptadores (DID, wallet, KMS, transport) y configuracion de agentes entre los tres servicios.

**Comportamiento actual:**
- issuer-service, holder-service y verifier-service tienen copias casi identicas de:
  - `custom-did-resolver.adapter.ts`, `custom-did-registrar.adapter.ts`
  - `webdid-resolver.adapter.ts`, `webdid-registrar.adapter.ts`
  - `internal-wallet.adapter.ts`, `external-wallet.adapter.ts`
  - `internal-kms.adapter.ts`, `external-kms.adapter.ts`
  - `ws-outbound-delayed-close.transport.ts`
- Cada servicio construye su agente manualmente con `new Agent(...)` y registra adaptadores de forma duplicada
- `ensureIssuerDid`, `ensureHolderDid`, `ensureVerifierDid` son practicamente identicos
- Lógica de invitacion OOB (create/receive) duplicada entre issuer, holder y verifier

**Comportamiento esperado:**
- Un solo paquete `@one/credo` (o `packages/credo-lib`) que exporte:
  - Adaptadores inyectables (sin leer env directamente)
  - Factory `createIssuerAgent`, `createHolderAgent`, `createVerifierAgent` con config unificada
  - `ensureDid` generico reutilizable
  - Helpers de invitacion OOB
- Los servicios migran a usar la libreria; dejan de duplicar codigo Credo
- Un solo camino de hacer las cosas: config → factory → agente listo

### Paso 6: Implementar agent factories (createIssuerAgent, createHolderAgent, createVerifierAgent)

**Descripcion:**
Crear las tres factories que encapsulen la construccion completa del agente: DependencyManager, registros, DidCommModule, protocolos segun rol.

**Archivos a modificar/crear:**
- `packages/credo/src/agent/create-issuer-agent.ts`
- `packages/credo/src/agent/create-holder-agent.ts`
- `packages/credo/src/agent/create-verifier-agent.ts`
- `packages/credo/src/agent/index.ts`

**Detalles de implementacion:**

```typescript
// create-holder-agent.ts
export async function createHolderAgent(
  config: CredoAgentBaseConfig,
  options?: { wsServer?: any }
): Promise<Agent> {
  const dm = new DependencyManager()
  registerWalletConfig(dm, config.wallet)
  registerKmsConfig(dm, config.kms)
  dm.registerInstance(
    DidCommModuleConfig,
    new DidCommModuleConfig({ endpoints: [config.didcommEndpoint] })
  )

  const inboundTransport = options?.wsServer
    ? new DidCommWsInboundTransport({ server: options.wsServer })
    : new DidCommWsInboundTransport({ port: /* configurar puerto por config? */ })

  const agent = new Agent({
    config: { label: config.label, /* ... */ },
    modules: {
      keyManagement: buildKeyManagementModule(config.kms),
      dids: buildDidsModule({ vdrServiceUrl: config.vdrServiceUrl, didcommEndpoint: config.didcommEndpoint }),
      w3cCredentials: new W3cCredentialsModule({}),
      didcomm: new DidCommModule({
        endpoints: [config.didcommEndpoint],
        transports: {
          inbound: [inboundTransport],
          outbound: [
            new DidCommHttpOutboundTransport(),
            new DidCommWsOutboundTransportDelayedClose(),
          ],
        },
        connections: { autoAcceptConnections: true },
        credentials: { credentialProtocols: [/* DidCommCredentialV2 + JsonLd */] },
        proofs: { proofProtocols: [/* DidCommProofV2 + DifPresentationExchange */] },
      }),
    },
    dependencies: agentDependencies,
  }, dm)

  await agent.initialize()
  return agent
}
```

- Issuer: solo credentials ( DidCommCredentialV2 + DidCommJsonLdCredentialFormatService )
- Holder: credentials + proofs
- Verifier: solo proofs

**Notas:**
- El puerto de DidCommWsInboundTransport puede pasarse en config (didcommPort) cuando no hay wsServer
- Los listeners de credenciales/pruebas (setupCredentialListeners, setupProofListeners) quedan en los servicios; la factory solo retorna el agente base

## Consideraciones Tecnicas

### Performance
- Los adaptadores hacen HTTP al vdr-service, wallet-service, kms-service; el impacto es el mismo que actualmente
- Una sola instancia de la libreria reduce duplicacion de codigo y tamaño de bundles

### Seguridad
- La config (URLs, walletId) se pasa explicitamente; no hay magic env
- Las claves siguen en KMS; la libreria no las manipula directamente

### Casos Edge
- Si vdr-service no responde, OneDidResolver retorna error de resolucion (igual que ahora)
- KeyManagementKeyNotFoundError: ensureDid limpia DidRecord y VDR, recrea DID (logica actual preservada)

### Compatibilidad
- Credo-TS: usar las mismas versiones (*) que los servicios actuales
- Wallet/KMS external vs internal: la libreria soporta ambos via config
